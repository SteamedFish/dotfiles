# 将想法转化为设计

## 概述

通过自然的协作对话，帮助将想法转化为完整的设计和规格说明。

首先了解当前项目的上下文，然后逐个提问以完善想法。一旦理解了要构建的内容，就以小块（200-300字）呈现设计，在每个部分之后检查是否看起来正确。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近的提交）
- **搜索现有实现** - 在提出新解决方案之前：
  - 在代码库中搜索类似功能
  - 寻找可复用的工具、辅助函数或模式
  - 检查其他模块是否解决了类似问题
  - 识别哪些可以扩展，哪些需要从头构建
- 逐个提问以完善想法
- 尽可能使用选择题，但开放式问题也可以
- 每条消息只提一个问题 - 如果一个话题需要更多探索，将其拆分为多个问题
- 重点理解：目的、约束条件、成功标准

**探索方案：**
- 提出 2-3 种不同的方案并权衡利弊
- 以对话方式呈现选项，包含你的建议和理由
- 首先展示你推荐的选项并解释原因

**呈现设计：**
- 一旦你认为理解了要构建的内容，就呈现设计
- 将其分成 200-300 字的小节
- 在每个部分之后询问到目前为止是否看起来正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果有不清楚的地方，准备好回头澄清

## 设计完成后

**文档：**
- 将经过验证的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely 技能
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问："准备好开始实现了吗？"
- 使用 using-git-worktrees 创建隔离的工作空间
- 使用 writing-plans 创建详细的实现计划

## 核心原则

- **一次只问一个问题** - 不要一次性提出多个问题让人应接不暇
- **优先使用选择题** - 在可能的情况下，比开放式问题更容易回答
- **严格遵循 YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方案
- **增量验证** - 分节呈现设计，逐节验证
- **保持灵活** - 当有不清楚的地方时，回头澄清
