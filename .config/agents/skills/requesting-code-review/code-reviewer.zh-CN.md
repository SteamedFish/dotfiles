## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否得当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否处理？

**架构：**
- 设计决策是否合理？
- 是否考虑可扩展性？
- 性能影响如何？
- 是否存在安全问题？

**测试：**
- 测试是否真正测试逻辑（而非模拟）？
- 边界情况是否覆盖？
- 需要时是否有集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否符合规范？
- 是否有范围蔓延？
- 破坏性变更是否有文档记录？

**生产就绪度：**
- 迁移策略（如有 schema 变更）？
- 是否考虑向后兼容性？
- 文档是否完整？
- 是否有明显 bug？

## 关键规则

**应该：**
- 按实际严重程度分类（并非所有都是严重）
- 具体说明（文件:行号，而非模糊）
- 解释问题为何重要
- 认可优点
- 给出明确结论

**不应该：**
- 不检查就说"看起来不错"
- 将吹毛求疵标记为严重
- 对未审查的代码给出反馈
- 模糊不清（"改进错误处理"）
- 避免给出明确结论
